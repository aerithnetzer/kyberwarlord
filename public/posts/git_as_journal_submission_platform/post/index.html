Git as a Journal Submission Platform

15.05.2024 00:00

I am currently attending the Library Publishing Forum in 2024, and a common theme of presentations that I have seen is managing publication workflows. Inspired by the Journal of Open Source Software, I have been thinking about how Git can facilitate a new model of journal submission and peer review.

The Steps of a Journal Publishing Workflows

In a traditional journal publishing workflow, the first step is an author submits a manuscript to a journal. Usually, this comes in a Docx format. The journal editor then sends the manuscript to reviewers, who provide feedback. The author then revises the manuscript and resubmits. The process continues until the manuscript is accepted by the editors or rejected.

Then, the manuscript is copyedited and formatted for publication. This includes formatting the article metadata, such as author names, affiliations, and abstracts according to the journal’s style guide and the Journal Article Tag Suite (JATS) standard. The manuscript is then published on the journal’s website and indexed in databases.

Git’s Role in Journal Publishing

This entire process, from submission to publication, mirrors the workflow of an open-source software development project. In FOSS projects, developers submit code changes/additions/fixes to a repository, which are then reviewed by maintainers. The maintainers provide feedback, and the developer revises the code. The process continues until the code is accepted or rejected.

Git can be used to manage the entire journal publishing workflow. Authors can submit manuscripts as merge requests to a Git repository. Reviewers can provide feedback as comments on the merge request. Authors can then revise the manuscript and resubmit.
